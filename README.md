# idea-spark · 灵感火花

> *「从资料和问题里点出小产品机会」*

<br>

**把书、文章、课程资料、聊天记录、客户抱怨和行业问题，转成一组有用有趣的小产品候选。**

不是给你一堆“灵感清单”让你自己挑。<br>
是先看资料里真实出现的痛点、流程和信息差，再生成候选、淘汰弱项、收束 Top 3。

## 装上就能用

打开你正在用的 AI agent，告诉它：

```text
帮我安装这个 skill：https://github.com/ghostyee2023/idea-spark
```

或者用通用 CLI 安装器：

```bash
npx skills add ghostyee2023/idea-spark
```

装好后直接说话：

```text
用 idea-spark 分析这篇文章，帮我挖 10 个小产品机会。
从这段客户聊天记录里找产品化机会，最后给我 Top 3。
我有一份课程资料，看看里面能不能提炼出工具、模板或服务型产品。
```

如果自动加载失败，手动把 `SKILL.md` 的内容粘贴进对话也能用。它本质就是一份 markdown + YAML frontmatter。

## 能做什么

灵感火花负责“从还不清楚的资料和问题里找机会”，它是三件套里的上游入口。

| 能力 | 说明 | 产出 |
| --- | --- | --- |
| **入口分流** | 判断输入是原始资料、模糊需求、已有 idea，还是候选清单 | 路由建议 |
| **机会地图** | 提炼痛点、重复流程、信息差、决策困难和可工具化节点 | 资料里的机会结构 |
| **多形态发散** | 生成工具、模板、插件、知识库、服务 MVP、诊断产品等候选 | Top 10 小产品机会 |
| **候选审计** | 保留候选、通过项、淘汰项和原因 | 不再反复捞弱想法 |
| **压力测试** | 检查 Top 3 是否真的不同、能不能构建、边界在哪里 | 更稳的 Top 3 |
| **结构化交接** | 用统一机会对象协议保留事实、推断、未知与验证状态 | 可直接交给 business-lens 的候选记录 |
| **source-pack** | 把可复用资料编译成机会资产包 | 可复用的资料机会包 |

## 怎么做的

灵感火花不靠“脑洞随机游走”，它做的是「来源信号 + 产品形态 + 候选淘汰」。

**先找来源信号**  
看资料里反复出现的痛点、重复动作、信息差、决策困难、人工流程和当前替代方案。

**再转成产品形态**  
同一个信号可能变成模板、插件、诊断工具、知识库、服务 MVP、工作流 Agent 或轻量脚本。

**然后淘汰弱候选**  
只会总结资料的，不算产品。没有重复输入输出的，不算产品。用户和场景不清楚的，先降级。

**最后收束 Top 3**  
不是挑最酷的，而是挑最有清晰用户、具体输入输出、可验证第一步的。

## 适合谁

- **内容创作者**：从文章、课程、热点里找产品化方向。
- **独立开发者**：从模糊问题里找小工具和插件机会。
- **咨询和交付团队**：从客户资料里找可产品化的交付资产。
- **知识库玩家**：把长期资料变成可复用的机会资产包。
- **课程设计者**：把资料转成模板、训练题、诊断工具和服务 MVP。

如果你已经有一个明确产品想法，灵感火花可能不是第一站。默认先用 `business-lens` 看商业逻辑；只有你明确只想做原型，并接受商业判断尚未完成时，才直接用 `maker-forge`。

## 三件套关系

```text
idea-spark      灵感火花：从资料和问题里找小产品机会
business-lens   商机透镜：判断商业逻辑、购买者、预算和获客
maker-forge     造物工坊：把通过判断的机会锻造成 3 小时 MVP
```

## 局限

诚实地说：

- **候选机会不是市场验证**：资料里出现痛点，不等于有人愿意付费。
- **发散不是越多越好**：真正有价值的是候选、淘汰和收束。
- **不替代商业判断**：有没有商机，要交给 `business-lens`。
- **不替代 MVP 设计**：第一版怎么做，要交给 `maker-forge`。
- **默认不写文件**：除非你明确要求保存、落地或生成资料包。

一个好的创意工具，不该让你沉迷点子。它应该帮你更快看见值得继续判断的机会。
